## 3.1 Phase 1: Explore & Record

**Goal:** Capture a causal trace of interaction with the website -- not just network requests, but the mapping between UI events and the API calls they trigger.

**Principle: Agent-first, human-fallback.** The meta-skill drives the browser itself using existing browser-use capabilities (accessibility tree, screenshots, DOM). A human only intervenes for actions the agent cannot perform (login with 2FA, CAPTCHA, selecting from a physical-world context the agent doesn't have, etc.).

**Runtime:** Playwright (headless or headed Chromium) controlled via CDP (Chrome DevTools Protocol), with an LLM agent layer on top for navigation decisions.

**What we capture:**

| Data | Source | Purpose |
|---|---|---|
| Agent/user events (click, submit, keypress) with timestamps | CDP `Input` domain, Playwright event hooks | Establish causality: which action triggered which request |
| HTTP request/response pairs (method, URL, headers, body) | CDP `Network.requestWillBeSent`, `Network.responseReceived`, `Network.getResponseBody` | Raw API surface |
| Request initiator (call stack) | CDP `Network.requestWillBeSent` `.initiator` field | Disambiguate: was this request triggered by user action or background polling? |
| Cookies, localStorage, sessionStorage | CDP `Storage` domain, `Network.requestWillBeSentExtraInfo` | Identify session/auth state dependencies |
| WebSocket frames | CDP `Network.webSocketFrameSent`, `Network.webSocketFrameReceived` | Capture real-time data channels |
| Page navigation & URL state | CDP `Page` domain | Map flows to URL patterns |
| Accessibility tree snapshots | Playwright `page.accessibility.snapshot()` | Agent's input for navigation decisions |
| Screenshots at key interaction points | Playwright `page.screenshot()` | Visual record for debugging and semantic analysis |

**Agent-Driven Exploration**

The meta-skill uses a browser-use agent to explore the site systematically. This is a two-layer setup:

```
Layer 1: Navigation Agent (LLM + browser-use)
  - Reads accessibility tree / screenshots
  - Decides what to click, type, navigate
  - Follows a task plan generated by the PM analysis
  - Can be powered by any browser-use framework:
    - Playwright + Claude directly (via MCP browser tool)
    - browser-use (open source, Python, LLM-driven)
    - Stagehand (Browserbase, TypeScript)
    - Or a custom thin wrapper over Playwright

Layer 2: Traffic Recorder (CDP, passive)
  - Runs underneath the navigation agent
  - Captures all network traffic, events, state
  - Transparent to the navigation agent -- it just browses normally
  - Produces C-HAR output
```

The navigation agent doesn't need to be sophisticated -- it just needs to exercise the site's main flows. Its task list comes from Step 0 below.

**Step 0: Task Planning**

Before browsing, the agent gets a high-level task plan:

1. Load the site's landing page. Read the accessibility tree / screenshot.
2. Identify the site's category (e-commerce, travel, social media, SaaS tool, etc.).
3. Use the knowledge base's domain intent library to generate a task list:
   - E-commerce: search for products, apply filters, view product details, add to cart, view cart, start checkout, look for coupons
   - Travel: search for flights/hotels, apply filters (dates, stops, price), view details, start booking
   - Social: browse feed, search, view profile, view post, create post, comment
4. For each task, generate 2-3 concrete parameter variations (e.g., search for "laptop", "headphones", "running shoes").

**Step 1: Agent Explores**

For each task in the plan:

```
Agent reads a11y tree / screenshot
  -> Decides action (click button, type in field, etc.)
  -> Executes via Playwright
  -> CDP recorder captures the resulting network traffic
  -> Agent reads updated a11y tree / screenshot
  -> Repeats until task is complete or agent is stuck
```

**Step 2: Human Handoff (only when needed)**

The agent hands off to the human when it encounters:

| Blocker | Agent's action |
|---|---|
| Login page (and no credentials provided) | Pause, open headed browser, ask human to log in. Resume recording after login. |
| CAPTCHA / challenge | Pause, show to human, resume after solved. |
| 2FA / SMS verification | Pause, ask human to complete, resume. |
| Ambiguous real-world choice (e.g., "pick your seat on the plane") | Pause, ask human to make the choice, resume. |
| Agent stuck for >N actions without progress | Pause, ask human to demonstrate the next steps. |

The handoff is minimal and bounded. The human completes the specific blocker, then control returns to the agent. The CDP recorder captures everything regardless of who is driving.

**Step 3: Variance Generation**

For flows the agent completed successfully, it re-runs them with different parameters to generate the variance needed for clustering:

```
Flow: "search for product"
  Run 1: search("laptop")           -> records traffic
  Run 2: search("headphones")       -> records traffic
  Run 3: search("running shoes")    -> records traffic

These 3 recordings let Phase 2 diff the requests and identify
which fields are user-input vs constant.
```

**Recording strategy summary:**

- The agent drives all exploration by default using a11y tree + screenshots for perception and Playwright for action.
- CDP runs passively underneath, capturing all traffic into C-HAR format.
- Human intervenes only for auth blockers and situations requiring real-world judgment.
- Each flow is repeated with parameter variations to enable clustering.

**Output:** A `recording/` directory containing C-HAR files, one per explored flow, with metadata indicating whether each flow was agent-driven or human-assisted.

## 3.1.5 Phase 1.5: Probe & Classify Execution Requirements

**Goal:** Empirically determine the minimum execution context each endpoint requires across multiple independent security dimensions.

The full security taxonomy, probing protocol, and execution mode derivation are documented in **[security-taxonomy.md](security-taxonomy.md)**.

**The Observation Paradox**

During Phase 1, all traffic is recorded inside a real browser with full cookies, TLS fingerprint, CSRF tokens, and JavaScript signing. Everything succeeds. You cannot determine from a set of successes which factors were *necessary* for success. For example:

- A request carries 20 cookies. Does the server check all 20, or just `session_id`?
- A request has an `X-CSRF-Token` header. Does the server enforce it for GET requests?
- The browser's TLS fingerprint (JA3/JA4) matches Chrome. Would the server reject a Python `requests` fingerprint?

The only way to know is to **try without and see what breaks**.

**Probing Protocol**

Probing tests each security dimension independently rather than walking a linear chain. See [security-taxonomy.md Section 3.1.5.5](security-taxonomy.md#3155-probing-protocol-revised) for the full revised probing protocol, which covers auth, CSRF, origin, TLS, and bot detection as separate probes.

**Probing is cheap:** Each endpoint needs a handful of HTTP requests. For a site with 20 endpoints, that's ~100-120 requests -- trivial compared to the recording phase.

**Observable signals** from Phase 1 recordings can **prioritize** probe order (but not replace probing):

| Signal from Recording | Heuristic | Confidence |
|---|---|---|
| `__cf_bm` / `_abck` cookies present | Cloudflare/Akamai bot detection likely | High that browser is needed |
| `X-CSRF-Token` in request headers | CSRF may be enforced on mutations | Medium |
| `Authorization: Bearer xxx` pattern | Token-based auth, may work without browser | Medium |
| High-entropy body fields (base64, hex) | Possible signing/encryption | Low (could just be UUIDs) |
| Simple GET with `Accept: application/json` | Likely works as direct HTTP | Medium |

The meta-skill's `knowledge/probe-heuristics.json` accumulates empirical data across sites: "of 47 sites with Cloudflare, direct HTTP failed for 44 (93.6%)." This prioritization improves over time (see [self-evolution.md Section 8](self-evolution.md#8-meta-skill-self-evolution)).

**Output:** A multi-dimensional security profile and derived execution strategy for each endpoint, stored in `api-map/security-profiles.json` (see [security-taxonomy.md Section 3.1.5.2](security-taxonomy.md#3152-security-profile-per-endpoint) for the full format). A simplified view is also exported to `api-map/execution-modes.json`:

```json
{
  "GET /api/search": {
    "mode": "session_replay",
    "security_profile_summary": "auth=cookie, csrf=none, bot=none",
    "probed_at": "2025-02-26T12:05:00Z"
  },
  "POST /api/cart/add": {
    "mode": "headless_browser",
    "security_profile_summary": "auth=cookie, csrf=required, bot=cloudflare_js",
    "probed_at": "2025-02-26T12:05:02Z"
  }
}
```

## 3.2 Phase 2: Analyze & Canonicalize

**Goal:** Transform raw recordings into a canonical API map -- a structured model of the website's API surface with parameterized templates and inferred schemas.

**Step A: Endpoint Clustering**

Group requests by `(HTTP method, URL path pattern, Content-Type, GraphQL operationName)`:

```
Concrete requests:
  GET /api/search?q=laptop&page=1&sort=price
  GET /api/search?q=phone&page=2&sort=rating
  GET /api/search?q=tablet&page=1&sort=relevance

Clustered endpoint:
  GET /api/search?q={query}&page={page}&sort={sort_by}
```

For REST APIs with path parameters:
```
  GET /api/products/B08N5WRWNW
  GET /api/products/B09V3KXJPB

Clustered:
  GET /api/products/{product_id}
```

Clustering algorithm:
1. Parse URLs into `(scheme, host, path_segments[], query_params{})`.
2. Group by `(method, path_segment_count, param_key_set)`.
3. Within each group, align path segments; segments that vary across samples become `{param}`.
4. For query parameters: keys that appear in all samples are required; values that vary are parameters; values that are constant are defaults/enums.

**Step B: Parameter Differentiation**

For each clustered endpoint, classify every variable field:

| Classification | Signal | Example |
|---|---|---|
| **User input** | Varies freely, maps to UI control | `q=laptop`, `destination=SFO` |
| **Pagination / cursor** | Monotonic, appears in sequence | `page=2`, `cursor=eyJhZG...` |
| **Session token** | Same across all requests in session, changes across sessions | `Cookie: session=abc123` |
| **CSRF / nonce** | Changes every request, often in hidden form fields or meta tags | `X-CSRF-Token: a8f3...` |
| **Derived / computed** | Looks like hash/timestamp, not user-controlled | `_t=1708900000`, `sig=hmac(...)` |
| **Constant** | Same value always | `format=json`, `v=2` |

Classification is done by a combination of heuristic rules (entropy analysis, pattern matching) and LLM-assisted semantic labeling (match field names to UI labels from the DOM).

**Step C: Schema Induction**

For each endpoint's request body and response body (when JSON):

1. **Structural inference:** Merge all observed JSON samples into a unified JSON Schema (types, required fields, array item schemas, nullable fields).
2. **Semantic annotation:** Use the LLM to generate human-readable field names and descriptions by correlating:
   - The JSON field name (`prdId` -> "Product ID")
   - The field value distribution (UUIDs, prices, dates, etc.)
   - The corresponding UI label from the DOM recording
   - The endpoint's role in the workflow

**Step D: Dependency Graph**

Map the causal chain between endpoints:

```
search_flights(origin, dest, date)
  -> response contains flight_offer_ids[]
    -> get_offer_details(offer_id)
      -> response contains fare_key
        -> create_booking(fare_key, passenger_info)
          -> response contains booking_id, payment_url
```

This is extracted from the C-HAR causality data: when response field X from endpoint A appears as request field Y in endpoint B, there is a data dependency `A.response.X -> B.request.Y`.

**Output:** An `api-map/` directory containing:
- `endpoints.json`: All clustered endpoints with parameter classifications
- `schemas/`: JSON Schema files for each endpoint's request/response
- `dependencies.json`: The inter-endpoint data flow graph

## 3.3 Phase 3: Generate Skill Package

**Goal:** Produce a deployable per-website skill package.

**Step A: Tool Synthesis**

Each semantically meaningful endpoint (or group of related endpoints) becomes a **tool**:

```json
{
  "name": "search_flights",
  "description": "Search for flights between two airports on a given date",
  "input_schema": {
    "type": "object",
    "properties": {
      "origin": { "type": "string", "description": "Origin airport IATA code (e.g., 'SFO')" },
      "destination": { "type": "string", "description": "Destination airport IATA code (e.g., 'JFK')" },
      "departure_date": { "type": "string", "format": "date", "description": "Departure date (YYYY-MM-DD)" },
      "passengers": { "type": "integer", "default": 1, "description": "Number of passengers" }
    },
    "required": ["origin", "destination", "departure_date"]
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "flights": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "offer_id": { "type": "string" },
            "airline": { "type": "string" },
            "price": { "type": "number" },
            "departure_time": { "type": "string" },
            "arrival_time": { "type": "string" },
            "stops": { "type": "integer" }
          }
        }
      }
    }
  },
  "security_profile": {
    "auth": { "mechanism": "cookie_session", "required_cookies": ["NID", "CONSENT"] },
    "session_protection": { "mechanism": "none" },
    "bot_detection": { "mechanism": "tls_fingerprint" },
    "human_verification": { "mechanism": "none" },
    "network": { "rate_limit": "unknown", "ip_bound": false },
    "request_integrity": { "mechanism": "protobuf_encoding" }
  },
  "execution_strategy": {
    "primary": "headless_browser",
    "needs_browser": true,
    "needs_headed": false,
    "needs_human": false,
    "determined_by": "probe",
    "fallback_chain": ["headless_browser", "headed_browser", "headed_browser_with_human"]
  },
  "execution": {
    "api": {
      "method": "GET",
      "url_template": "https://www.google.com/travel/flights/api/search?origin={origin}&dest={destination}&date={departure_date}&pax={passengers}",
      "headers": { "Accept": "application/json" },
      "csrf_extractor": null
    },
    "ui_fallback": {
      "steps": [
        "Navigate to https://www.google.com/travel/flights",
        "Type {origin} in 'Where from?' field",
        "Type {destination} in 'Where to?' field",
        "Select {departure_date} from date picker",
        "Click 'Search'"
      ]
    }
  }
}
```

Key design decisions:
- Every tool carries a **multi-dimensional security profile** (from Phase 1.5 probing) and a derived **execution strategy**.
- Every tool has **both** an API execution path and a UI fallback path.
- The execution strategy specifies the primary mode and a fallback chain. The fallback chain only contains modes *above* the primary (more capable, more expensive).
- Input/output schemas are in JSON Schema format, consumable by any LLM tool-use framework.

**Step B: Workflow Composition**

Common user tasks are represented as workflow DAGs:

```yaml
# workflows/book_cheapest_flight.yaml
name: book_cheapest_flight
description: Find and book the cheapest flight with applicable coupons
steps:
  - id: search
    tool: search_flights
    inputs:
      origin: $input.origin
      destination: $input.destination
      departure_date: $input.date

  - id: pick_cheapest
    tool: sort_and_select
    inputs:
      items: $search.output.flights
      sort_by: price
      order: asc
      select: first

  - id: check_coupons
    tool: list_available_coupons
    inputs:
      offer_id: $pick_cheapest.output.offer_id

  - id: apply_best_coupon
    tool: apply_coupon
    condition: $check_coupons.output.coupons.length > 0
    inputs:
      offer_id: $pick_cheapest.output.offer_id
      coupon_code: $check_coupons.output.best_coupon.code

  - id: create_booking
    tool: create_booking
    inputs:
      offer_id: $pick_cheapest.output.offer_id
      passenger_info: $input.passenger_info
    requires_confirmation: true  # human-in-the-loop before payment
```

**Step C: Verifier Generation**

Each tool gets an automated verifier that checks execution success:

```javascript
// verifiers/search_flights.js
module.exports = {
  verify(response) {
    return {
      success: response.status === 200
        && Array.isArray(response.body?.flights)
        && response.body.flights.length > 0,
      error: response.status !== 200
        ? `HTTP ${response.status}`
        : 'No flights returned'
    };
  }
};
```

**Step D: Fingerprinting**

Compute version fingerprints for change detection:

```json
{
  "site": "google.com/travel/flights",
  "captured_at": "2025-02-26T12:00:00Z",
  "fingerprints": {
    "js_bundle_hash": "sha256:a1b2c3...",
    "api_endpoint_set_hash": "sha256:d4e5f6...",
    "search_response_shape_hash": "sha256:789abc...",
    "graphql_schema_hash": null
  }
}
```

## 3.4 Phase 4: Execute, Verify & Self-Heal

**Tiered Execution Model**

Each tool's execution mode was empirically determined during Phase 1.5 (Probing) across multiple security dimensions. The runtime uses the classified mode as the starting point, falling back up the chain on failure:

| Mode | Browser? | Headed? | Human? | Satisfies |
|---|---|---|---|---|
| `direct_http` | No | No | No | Auth: none/apikey/bearer. No bot detection. No CSRF. |
| `session_replay` | No | No | No | Auth: cookie. No CSRF enforcement. No bot detection. |
| `session_replay_with_csrf` | No | No | No | Auth: cookie + CSRF. Bot detection: none or UA-only. |
| `headless_browser` | Yes | No | No | Bot: JS challenge, TLS fingerprint. Integrity: JS signing. |
| `headed_browser` | Yes | Yes | No | Bot: headless detection, CDP detection, behavioral. |
| `headed_browser_with_human` | Yes | Yes | Yes | Human: CAPTCHA, 2FA, hardware key, payment confirmation. |

See [security-taxonomy.md Section 3.1.5.4](security-taxonomy.md#3154-revised-execution-modes) for the full derivation of these modes from the security taxonomy.

**Why tiered matters:**
- `direct_http` is **10-100x faster** than `headless_browser` and requires no browser process.
- Many read operations (search, browse, detail) on many sites work at `direct_http` or `session_replay`.
- `headless_browser` is the "usually works" fallback, but now we only use it when empirically necessary.
- `headed_browser` handles the gap between headless and human-assisted -- some sites reject headless Playwright but accept headed Playwright without human interaction.
- `headed_browser_with_human` is the last resort -- functionally identical to a traditional browser-use agent, but scoped to specific operations rather than the entire interaction.

**For `headless_browser` and `headed_browser` execution:** inject a small JavaScript bridge into the page via Playwright's `page.evaluate()`. The bridge:
- Reads the current CSRF token / session state from the DOM / cookies / localStorage
- Constructs the request using the tool's template + current state + user inputs
- Executes via `fetch()` within the page context
- Returns the parsed JSON response

```
Agent                    Browser (Playwright)             Website Server
  |                            |                              |
  |-- call_tool(search, args) -|                              |
  |                            |-- page.evaluate(bridge.js) --|
  |                            |   fetch('/api/search', {     |
  |                            |     headers: {csrf, cookie}  |
  |                            |     body: templated(args)    |
  |                            |   })                         |
  |                            |                   <--------- |  JSON response
  |                            |-- structured result -------->|
  |<-- tool output (JSON) ----|                              |
```

**Fallback Cascade**

When execution at the classified mode fails, the runtime walks up the mode chain:

```
Execute at classified mode (e.g., direct_http)
   |-- Success? -> return structured result
   |-- Failure (HTTP error, shape mismatch, timeout)?
       |
Escalate to next mode (e.g., session_replay)
   |-- Success? -> return result + log escalation for investigation
   |-- Failure?
       |
Continue escalating: session_replay_with_csrf -> headless_browser -> headed_browser
   |-- headed_browser fails?
       |
Final attempt: headed_browser_with_human (UI automation with human handoff)
   |-- Still fails?
       |
Trigger self-heal (re-probe + re-record this flow)
```

**Important:** A mode escalation that succeeds is a signal that the probing classification may be stale. The system logs these events, and if a tool consistently escalates beyond its classified mode, the skill is re-probed and the execution strategy is updated.

**Self-Healing Pipeline**

When a skill's tools start failing:

1. **Detect:** Verifier returns `success: false`. Compare current site fingerprint against stored fingerprint -- if different, the site has changed.
2. **Re-record:** Run the exploration flow again for the failing tool(s) only.
3. **Diff:** Compare new recording against stored templates:
   - Endpoint URL changed?
   - Request/response fields renamed or restructured?
   - New required parameters added?
   - Authentication flow changed?
4. **Patch:** Auto-update the tool's URL template, schema, and extractors.
5. **Test:** Run the tool's test suite against the patched definition.
6. **Publish:** If tests pass, commit the updated skill. If not, flag for human review.

This treats the website as a **dependency with an unversioned API** and applies the same contract-testing discipline used for microservice APIs.
