# Review: qi_note.md

## TL;DR
这份笔记抓住了正确方向：`agent-first`、`human-fallback`、`用网站构建过程反哺 meta-skill`。建议把想法转成可执行路线图，避免一次性铺太大。

## 主要问题与建议

### P1: “先做100个站点”节奏过激进
- 风险是样本偏差、成本失控、迭代反馈过慢。
- 建议阶段化：10 -> 30 -> 100，每阶段先复盘再扩量。

### P1: 需要先定义站点采样策略
- 若 100 个站点集中在同一类型，学习会过拟合。
- 建议按垂直和防护强度分层抽样（内容站、SaaS、电商、旅行、社交）。

### P1: `api-key-skill` 是好方向，但应作为独立产品线
- 注册/获取 API key 往往涉及 ToS、邮箱验证、风控流程。
- 建议设置 allowlist，只覆盖明确允许自动化注册的平台。

### P2: “免费模型 bootstrap”需要稳定性与合规评估
- 免费接口可用性波动大，且条款可能变化。
- 建议只作为开发期后备路径，不作为核心依赖。

## KISS 落地版本
1. 先做 `agent-first explore&record` + `human handoff` 的最小闭环。
2. 在 10 个站点上验证：成功率、耗时、人工介入次数。
3. 再决定是否启动 `api-key-skill` 子项目。

## 结论
笔记中的方向是对的，关键是把“愿景”压缩成“可量化、可分阶段执行”的计划。
