## Per-Website Skill Package Format

### Minimal Package Structure

```
google-flights/
├── manifest.json         # Metadata + dependency graph + fingerprint
├── openapi.yaml          # OpenAPI 3.1 + x-web-skill extensions (canonical output)
├── extractors/           # JS snippets for complex token extraction (optional)
│   └── csrf.js
└── tests/                # Recorded input → expected output shape
    └── search_flights.test.json
```

**4 items, not 10.** Every directory earns its existence by being necessary for the first working skill.

**Default: single file per site.** One `openapi.yaml` shares `servers`, `info.x-web-skill` (fingerprint etc.), and `components/schemas` across operations. For sites with many operations (100+), OpenAPI's `$ref` mechanism supports splitting into multiple files. For MVP, single file is sufficient.

**Multi-domain sites** (e.g., `amazon.com` + `api.amazon.com`): handled via per-operation `servers` overrides in OpenAPI. All operations stay in one spec regardless of backend domains.

**What was eliminated and why:**
- `SKILL.md` — not a compiler output. Generated by `web-skill install` when deploying to an agent workspace. See [compiler-output-and-runtime.md](compiler-output-and-runtime.md) §2.5.
- `tools/` directory — replaced by `openapi.yaml`. One standard spec per site instead of one custom JSON per tool. Eliminates the custom tool definition format and its parameter duplication (`body_template` vs `parameters`).
- `templates/` — redundant with OpenAPI's native path/requestBody definitions
- `verifiers/` — response schema in OpenAPI validates responses
- `workflows/` — LLM agent plans from operation descriptions + dependency graph
- `fingerprints/` directory — a field in `manifest.json` (and `info.x-web-skill.fingerprint` in OpenAPI)
- `bridge/` — runtime code in the CLI executor, not skill-specific
- `wrappers/` — generated on demand by CLI export/install commands

### Interoperability Model: OpenAPI as Single Source of Truth

The package is canonical around `openapi.yaml`. All other formats are generated from this source of truth:

1. **CLI navigation** (primary): `web-skill <site>` reads the OpenAPI spec and presents operations progressively to minimize agent context usage.
2. **CLI execution** (primary): `web-skill <site> exec <tool> '{...}'` reads the OpenAPI operation, constructs the HTTP request, handles session/CSRF/browser context, and returns JSON.
3. **SKILL.md generation** (on install): `web-skill install <site>` generates a SKILL.md in the target agent's workspace.
4. **MCP adapter** (optional): wraps the same executor for agents without shell access.
5. **Provider-specific tool schemas** (on demand): mechanical extraction from OpenAPI operations to OpenAI/Anthropic/Gemini/MCP formats (~10 lines each).

The OpenAPI spec is the single source of truth. If any derived format disagrees with `openapi.yaml`, `openapi.yaml` wins.

### manifest.json

```json
{
  "name": "google-flights-web-skill",
  "version": "1.0.0",
  "spec_version": "0.1.0",
  "site": "google.com/travel/flights",
  "generated_at": "2026-02-27T12:00:00Z",
  "requires_auth": true,
  "fingerprint": {
    "js_bundle_hash": "sha256:a1b2c3...",
    "api_endpoint_set_hash": "sha256:d4e5f6...",
    "last_validated": "2026-02-27T12:00:00Z"
  },
  "dependencies": {
    "search_flights.flights[].offer_id": "get_flight_details.offer_id",
    "get_flight_details.fare_key": "create_booking.fare_key"
  }
}
```

Design principles:
- No derived fields (tool_count, capability lists). Derive from `openapi.yaml` at load time.
- `spec_version` field supports future format migration.
- `dependencies` encodes the inter-tool data flow graph — sufficient for LLM agents to plan multi-step sequences.

### OpenAPI Spec (Canonical Format)

Each operation in `openapi.yaml` contains everything needed to understand and execute it. The format uses standard OpenAPI 3.1 with `x-web-skill` vendor extensions for runtime metadata. See [compiler-output-and-runtime.md](compiler-output-and-runtime.md) §2.1 for the full spec with examples.

Example operation:

```yaml
/travel/flights/api/search:
  post:
    operationId: search_flights
    summary: Search for flights between two airports on a given date
    x-web-skill:
      mode: session_http
      human_handoff: false
      session:
        csrf: "document.querySelector('meta[name=csrf]').content"
    requestBody:
      required: true
      content:
        application/json:
          schema:
            type: object
            properties:
              origin:
                type: string
                description: "Origin airport IATA code (e.g., 'SFO')"
              dest:
                type: string
                description: "Destination airport IATA code (e.g., 'JFK')"
              date:
                type: string
                format: date
                description: "Departure date (YYYY-MM-DD)"
              passengers:
                type: integer
                default: 1
            required: [origin, dest, date]
    responses:
      "200":
        description: Flight search results
        content:
          application/json:
            schema:
              type: object
              properties:
                flights:
                  type: array
                  items:
                    type: object
                    properties:
                      offer_id: { type: string }
                      airline: { type: string }
                      price: { type: number }
                      stops: { type: integer }
```

Key points:
- **Standard format**: OpenAPI 3.1 — no custom tool definition format to learn.
- **Zero parameter duplication**: `requestBody.schema` IS the request parameter schema. No `body_template` mapping layer.
- **Agent-facing info** (`operationId`, `summary`, parameter schemas, response schemas): what the agent sees via CLI navigation. Mechanically extractable to any LLM provider format.
- **Execution-facing info** (`x-web-skill.*`): what the CLI executor uses. The agent never reads these directly.
- **Response validation**: the OpenAPI response schema validates responses — no separate verifier needed.
- **Deterministic execution only**: OpenAPI operations are atomic API calls. UI automation procedures (browser-use-style step-by-step instructions) are a separate artifact.
- **Session config**: CSRF extractors inline in `x-web-skill.session` for simple cases. For complex multi-step extraction, reference a file in `extractors/`.

**Deriving LLM tool schemas** (mechanical, ~10 lines each):

| Target | Transformation |
|---|---|
| OpenAI | Extract `operationId` -> `name`, `summary` -> `description`, `requestBody.schema` -> `parameters` |
| Anthropic | Same extraction, rename to `input_schema` |
| MCP | Same extraction, rename to `inputSchema`, map response schema -> `outputSchema` |
| Gemini | Same extraction |

### SKILL.md (Generated on Install, Not by Compiler)

SKILL.md is generated by `web-skill install <site>` when deploying to a target agent's workspace. It follows the Agent Skills standard (agentskills.io):

```yaml
---
name: google-flights
description: >
  Search flights, check prices, and get fare details on Google Flights.
  Use when the user needs flight information, price comparisons, or booking details.
allowed-tools: Bash(web-skill:*)
---
```

```markdown
# Google Flights

## Tools
- `search_flights(origin, dest, date)` — Search flights
- `get_flight_details(offer_id)` — Fare details for a flight
- `price_graph(origin, dest, month)` — Monthly price calendar

## Usage
web-skill google-flights search_flights '{"origin":"SFO","destination":"JFK","departure_date":"2026-04-01"}'

## Discovery
Run `web-skill google-flights` to see all tools with parameters.
Run `web-skill google-flights <tool>` to see detailed parameter info.
```

~15 lines. The agent reads this, knows what's available, knows how to call the CLI. Detailed parameter info is discovered progressively via CLI commands, not loaded upfront.

**Not in the compiler output** because:
- SKILL.md is an agent-ecosystem artifact, not a data artifact
- Different agents may use different workspace paths (`.claude/skills/`, `.cursor/skills/`, etc.)
- The `web-skill install` command handles path resolution per agent

## The Meta-Skill: web-skill

The meta-skill is invoked by the user to generate a new per-website skill.

### SKILL.md (User-Facing)

```yaml
---
name: web-skill
description: >
  Generate a structured API skill for any website. Use when the user wants to
  create tools for interacting with a website via API instead of browser automation.
allowed-tools: Bash(web-skill:*) Bash(npx:*)
---
```

```markdown
# web-skill

## Usage
web-skill compile https://www.google.com/travel/flights

## What it does
1. Opens the site and explores its functionality (agent-driven)
2. Extracts the underlying API endpoints from network traffic
3. Probes each endpoint to find the cheapest execution mode
4. Generates typed OpenAPI operations with execution mode annotations
5. Tests the tools and produces a ready-to-use skill package

## Output
A skill package at `~/.web-skill/sites/<site>/`

## Install to agent workspace
web-skill install <site>
```

## Execution Runtime

### CLI Architecture

```
Agent (Claude Code, Codex, Cursor, etc.)
    |
    |-- bash: web-skill <site> <command> -->  CLI binary (Node.js)
                                                |
                                                |-- Spec Navigator
                                                |     reads openapi.yaml
                                                |     progressive disclosure
                                                |
                                                |-- Executor
                                                |     |-- direct_http (node-fetch)
                                                |     |-- session_http (cookies + CSRF)
                                                |     |-- browser_fetch (Playwright page.evaluate)
                                                |     |-- fallback cascade
                                                |     |-- response verification
                                                |
                                                |-- Skill Registry
                                                |     ~/.web-skill/sites/
                                                |       google-flights/
                                                |       amazon/
                                                |       ...
                                                |
                                                |-- Session Manager (background daemon)
                                                      browser sessions, cookie jars
                                                      auto-start on first exec, auto-exit on idle
```

### CLI Commands

**Navigation (spec browsing):**

| Command | Description | Token cost |
|---|---|---|
| `web-skill sites` | List all installed sites | ~50 tokens |
| `web-skill <site>` | List tools for a site (name + one-liner) | ~100-200 tokens |
| `web-skill <site> <tool>` | Show tool parameters + returns summary | ~100-200 tokens |
| `web-skill <site> <tool> --full` | Show complete tool definition | ~300-500 tokens |
| `web-skill <site> <tool> --param <name>` | Show one parameter in detail | ~30-50 tokens |

**Execution:**

| Command | Description |
|---|---|
| `web-skill <site> exec <tool> '<json>'` | Execute a tool, return JSON to stdout |
| `web-skill <site> status` | Check site health (fingerprint match, test status) |
| `web-skill <site> heal` | Trigger self-healing for a site |

**Management:**

| Command | Description |
|---|---|
| `web-skill compile <url>` | Run the compiler pipeline for a new site |
| `web-skill install <site>` | Generate SKILL.md in target agent workspace |
| `web-skill export openapi <site>` | Export standalone OpenAPI 3.1 spec (without x-web-skill extensions) |
| `web-skill export <format> <site>` | Export to provider format (openai, anthropic, mcp, gemini) |

### MCP Adapter (Optional)

For agents without shell access, a thin MCP server wraps the same executor:

```bash
# Start MCP server (stdio transport)
web-skill mcp-serve
```

The MCP server dynamically registers tools from all installed sites. It uses the same executor code path as the CLI — no forked logic.

### Skill Discovery

- Discovery: file system scan of `~/.web-skill/sites/` at startup
- Hot-reloading: watch for new skill directories, register tools without restart
- Conflict: if two skills target the same site, use the newer version

### Versioning (MVP)

Version is a monotonic counter (1.0.0, 1.1.0, ...). No semver infrastructure until needed. Version increments whenever the skill is regenerated or self-healed.
